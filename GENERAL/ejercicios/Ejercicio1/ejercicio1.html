Introducción

SEO (optimización de motor de búsqueda) es la práctica de optimizar un sitio web o página web para aumentar la cantidad y la calidad de su tráfico de los resultados orgánicos de un motor de búsqueda.

El SEO es más popular que nunca y ha experimentado un gran crecimiento en los últimos años a medida que las empresas buscaban nuevas formas de llegar a los clientes. La popularidad del SEO ha superado con creces a otros canales digitales.

Figura 8.1 Comparación de Google Trends de SEO versus pago por clic, marketing en redes sociales y marketing por correo electrónico.

El propósito del capítulo de SEO del Web Almanac es analizar varios elementos relacionados con la optimización de un sitio web. En este capítulo, verificaremos si los sitios web brindan una excelente experiencia para los usuarios y los motores de búsqueda.

Se utilizaron muchas fuentes de datos para nuestro análisis, incluido Lighthouse, el Informe de experiencia del usuario de Chrome (CrUX) , así como elementos HTML sin procesar y renderizados del Archivo HTTP en dispositivos móviles y computadoras de escritorio. En el caso de HTTP Archive y Lighthouse, los datos se limitan a los datos identificados en las páginas de inicio de los sitios web únicamente, no a los rastreos de todo el sitio. Tenga esto en cuenta al sacar conclusiones de nuestros resultados. Puede obtener más información sobre el análisis en nuestra página de Metodología.

Siga leyendo para obtener más información sobre el estado actual de la web y su compatibilidad con los motores de búsqueda.

Capacidad de rastreo e indexabilidad

Para devolver resultados relevantes a estas consultas de los usuarios, los motores de búsqueda deben crear un índice de la web. El proceso para eso implica:

  1. Rastreo: los motores de búsqueda utilizan rastreadores web, o arañas, para visitar páginas en Internet. Encuentran nuevas páginas a través de fuentes como mapas de sitios o enlaces entre páginas.
  2. Procesamiento: en este paso, los motores de búsqueda pueden representar el contenido de las páginas. Extraerán la información que necesitan, como contenido y enlaces, que utilizarán para crear y actualizar su índice, clasificar páginas y descubrir contenido nuevo.
  3. Indexación: las páginas que cumplen con ciertos requisitos de indexabilidad en torno a la calidad y singularidad del contenido normalmente se indexarán. Estas páginas indexadas son elegibles para ser devueltas para consultas de usuarios.

Analicemos algunos problemas que pueden afectar la capacidad de rastreo y la indexación.

robots.txt

497 / 5000 Resultados de traducción robots.txt es un archivo ubicado en la carpeta raíz de cada subdominio en un sitio web que le dice a los robots, como los rastreadores de motores de búsqueda, dónde pueden y dónde no pueden ir.

El 81,9% de los sitios web utilizan el archivo robots.txt (móvil). En comparación con años anteriores (72,2% en 2019 y 80,5% en 2020), esa es una ligera mejora.

Tener un archivo robots.txt no es un requisito. Si devuelve un 404 no encontrado, Google supone que se pueden rastrear todas las páginas de un sitio web. Otros motores de búsqueda pueden tratar esto de manera diferente.

Figura 8.4. Distribución del tamaño de robots.txt.

El uso de robots.txt permite a los propietarios de sitios web controlar los robots de los motores de búsqueda. Sin embargo, los datos mostraron que hasta el 16,5% de los sitios web no tienen un archivo robots.txt.

Los sitios web pueden tener archivos robots.txt mal configurados. Por ejemplo, algunos sitios web populares estaban (presumiblemente por error) bloqueando los motores de búsqueda. Google puede mantener estos sitios web indexados durante un período de tiempo, pero eventualmente su visibilidad en los resultados de búsqueda disminuirá.

Otra categoría de errores relacionados con el archivo robots.txt es la accesibilidad y / o los errores de red, lo que significa que el archivo robots.txt existe pero no se puede acceder a él. Si Google solicita un archivo robots.txt y obtiene un error de este tipo, el bot puede dejar de solicitar páginas por un tiempo. La lógica detrás de esto es que los motores de búsqueda no están seguros de si una página determinada puede o no ser rastreada, por lo que espera hasta que se pueda acceder a robots.txt.

~ 0.3% de los sitios web en nuestro conjunto de datos arrojaron 403 Forbidden o 5xx. Diferentes bots pueden manejar estos errores de manera diferente, por lo que no sabemos exactamente qué pudo haber visto Googlebot.

La última información disponible de Google, de 2019, es que hasta el 5% de los sitios web mostraban temporalmente 5xx en robots.txt, mientras que hasta el 26% eran inaccesibles.

Figura 8.3. Desglose de los códigos de estado de robots.txt que encontró el robot de Google.

Dos cosas pueden causar la discrepancia entre el Archivo HTTP y los datos de Google:

  1. Google presenta datos de hace 2 años, mientras que HTTP Archive se basa en información reciente, o
  2. HTTP Archive se centra en sitios web que son lo suficientemente populares como para incluirse en los datos de CrUX, mientras que Google intenta visitar todos los sitios web conocidos.

Etiquetas canónicas

La web es un conjunto masivo de documentos, algunos de los cuales son duplicados. Para evitar problemas de contenido duplicado, los webmasters pueden usar etiquetas canónicas para indicar a los motores de búsqueda qué versión prefieren indexar. Canonicals también ayuda a consolidar señales como enlaces a la página de clasificación.

Figura 8.7 Uso canónico.

Los datos muestran una mayor adopción de etiquetas canónicas a lo largo de los años. Por ejemplo, la edición de 2019 muestra que el 48,3% de las páginas móviles usaban una etiqueta canónica. En la edición de 2020, el porcentaje creció al 53,6%, y en 2021 vemos el 58,5%.

Más páginas móviles tienen configurados canónicos que sus contrapartes de escritorio. Además, el 8,3% de las páginas móviles y el 4,3% de las de escritorio se canonizan a otra página para que den una pista clara a Google y otros motores de búsqueda de que la página indicada en la etiqueta canónica es la que debería indexarse.

Un mayor número de páginas canonicalizadas en dispositivos móviles parece estar relacionado con sitios web que utilizan URL móviles independientes. En estos casos, Google recomienda colocar una etiqueta rel = "canonical" que apunte a las URL de escritorio correspondientes.

Nuestro conjunto de datos y análisis se limitan a las páginas de inicio de los sitios web; Es probable que los datos sean diferentes al considerar todas las URL de los sitios web probados.

Dos métodos para implementar etiquetas canónicas

Al implementar canónicos, existen dos métodos para especificar etiquetas canónicas:

  • En la sección <head> de HTML de una página
  • En los header HTML (a través del encabezado Link HTTP)

Explore los resultados Muestre la descripción de la Figura 8.8 Figura 8.8. Uso canónico sin procesar versus renderizado. Implementar etiquetas canónicas en el <head> de una página HTML es mucho más popular que usar el método de encabezado Link. La implementación de la etiqueta en la sección principal generalmente se considera más fácil, por lo que ese uso es mucho mayor.

También vimos un ligero cambio <1%) en lo canónico entre el HTML sin procesar entregado y el HTML renderizado después de que se haya aplicado JavaScript.

Etiquetas canónicas en conflicto

A veces, las páginas contienen más de una etiqueta canónica. Cuando hay señales contradictorias como esta, los motores de búsqueda tendrán que resolverlo. Uno de los defensores de las búsquedas de Google, Martin Splitt, dijo una vez que provoca un comportamiento indefinido por parte de Google.

La figura anterior muestra que hasta el 1.3% de las páginas móviles tienen diferentes etiquetas canónicas en el HTML inicial y la versión renderizada.

El capítulo del año pasado señaló que "se puede encontrar un conflicto similar con los diferentes métodos de implementación, con un 0.15% de las páginas móviles y un 0.17% de las de escritorio que muestran conflictos entre las etiquetas canónicas implementadas a través de sus encabezados HTTP y HTML".

Los datos de este año sobre ese conflicto son aún más preocupantes. Las páginas envían señales contradictorias en el 0,4% de los casos en computadoras de escritorio y en el 0,3% de los casos en dispositivos móviles.

Como los datos de Web Almanac solo se ven en las páginas de inicio, puede haber problemas adicionales con las páginas ubicadas más profundamente en la arquitectura, que son páginas que probablemente necesiten señales canónicas.

2021 vio un mayor enfoque en la experiencia del usuario. Google lanzó la Actualización de experiencia de página que incluía señales existentes, como HTTPS y compatibilidad con dispositivos móviles, y nuevas métricas de velocidad llamadas Core Web Vitals.

Compatibilidad con dispositivos móviles

Este año se ha producido un ligero repunte en la compatibilidad con dispositivos móviles. Las implementaciones de diseño receptivo han aumentado, mientras que el servicio dinámico se ha mantenido relativamente plano.

El diseño receptivo envía el mismo código y ajusta la forma en que se muestra el sitio web en función del tamaño de la pantalla, mientras que el servicio dinámico envía un código diferente según el dispositivo. La metaetiqueta de la ventana gráfica se utilizó para identificar los sitios web receptivos frente al encabezado Vary: User-Agent para identificar los sitios web que utilizan el servicio dinámico.

Elementos fundamentales de la Web

Core Web Vitals son las nuevas métricas de velocidad que forman parte de las señales de Page Experience de Google. Las métricas miden la carga visual con Largest Contentful Paint (LCP), la estabilidad visual con Cumulative Layout Shift (CLS) y la interactividad con First Input Delay (FID).

Los datos provienen del Informe de experiencia del usuario de Chrome (CrUX), que registra datos del mundo real de los usuarios de Chrome que optaron por participar.

Figura 8.13. Tendencia de métricas básicas de web vitals.

El 29% de los sitios web móviles están superando los umbrales de Core Web Vitals, frente al 20% del año pasado. La mayoría de los sitios web están aprobando la FID, pero los propietarios de sitios web parecen tener dificultades para mejorar CLS y LCP. En la página

Los motores de búsqueda analizan el contenido de su página para determinar si es un resultado relevante para la consulta de búsqueda. Otros elementos de la página también pueden afectar la clasificación o la apariencia en los motores de búsqueda.

Metadatos

Los metadatos incluyen elementos <title> y etiquetas <meta name = "description">. Los metadatos pueden afectar directa y / o indirectamente al rendimiento de SEO.

Recuento de palabras

El número de palabras en una página no es un factor de clasificación, pero la forma en que las páginas entregan palabras puede afectar profundamente las clasificaciones. Las palabras pueden estar en el código de la página sin procesar o en el contenido renderizado.

Recuento de palabras renderizadas

Primero, miramos el contenido de la página renderizada. Renderizado es el contenido de la página después de que el navegador haya ejecutado todo JavaScript y cualquier otro código que modifique el DOM o CSSOM.

Figura 8.24. Palabras visibles representadas por percentil.

  • La mediana de la página de escritorio renderizada contiene 425 palabras, frente a las 402 palabras de 2020.
  • La mediana de la página móvil renderizada contiene 367 palabras, frente a las 348 palabras de 2020.
  • Las páginas móviles renderizadas contienen un 13,6% menos de palabras que las páginas de escritorio renderizadas. Ten en cuenta que Google es un índice solo para dispositivos móviles. Es posible que el contenido que no esté en la versión móvil no se indexe.

Datos estructurados

Históricamente, los motores de búsqueda han trabajado con datos no estructurados: las pilas de palabras, párrafos y otros contenidos que componen el texto de una página.

El marcado de esquema y otros tipos de datos estructurados proporcionan a los motores de búsqueda otra forma de analizar y organizar el contenido. Los datos estructurados impulsan muchas de las funciones de búsqueda de Google.

Al igual que las palabras en la página, los datos estructurados se pueden modificar con JavaScript.

Figura 8.26. Uso de datos de estructura.

El 42,5% de las páginas móviles y el 41,8% de las páginas de escritorio tienen datos estructurados en el HTML. JavaScript modifica los datos estructurados en el 4,7% de las páginas móviles y el 4,5% de las páginas de escritorio.

En el 1,7% de las páginas móviles y el 1,4% de las páginas de escritorio, JavaScript agrega datos estructurados donde no existían en la respuesta HTML inicial.

Formatos de datos estructurados más importantes

Figura 8.27. Desglose de formatos de datos estructurados.

Hay varias formas de incluir datos estructurados en una página: JSON-LD, microdatos, RDFa y microformatos2. JSON-LD es el método de implementación más popular. Más del 60% de las páginas de escritorio y móviles que tienen datos estructurados lo implementan con JSON-LD.

Entre los sitios web que implementan datos estructurados, más del 36% de las páginas de escritorio y móviles usan microdatos y menos del 3% de las páginas usan RDFa o microformatos2.

La adopción de datos estructurados ha aumentado un poco desde el año pasado. Se utiliza en el 33,2% de las páginas en 2021 frente al 30,6% en 2020.

Conclusión

Los sitios web están mejorando lentamente desde una perspectiva de SEO. Probablemente debido a una combinación de sitios web que mejoran su SEO y las plataformas que alojan sitios web también mejoran. La web es un lugar grande y desordenado, por lo que todavía hay mucho que hacer, pero es bueno ver un progreso constante.
